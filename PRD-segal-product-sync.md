# מסמך דרישות — ייבוא וסנכרון מוצרי סגל ל-Shopify

> **מטרה:** להגדיר, שדה-אחר-שדה, איך מוצר מאתר הספק **סגל** (segalbaby.co.il) מיובא ל-Shopify ומסונכרן.
> **חלק ממערכת Supplier Product Sync.** אומת מול החנות החיה על 4 סוגי מוצרים (מעקה/ארון/מיטה/שידה), שדה-שדה.
> **מחיר:** מייבאים את **מחיר הספק כמו שהוא** ל-`variant.price`. המשתמש משנה אח"כ עם הנוסחה שלו (לא חוסם ייבוא).

---

## 0. מקור הנתונים — היברידי (API + דף HTML)
לסגל **WooCommerce Store API ציבורי**, אבל **המפרט/תוכן הלשוניות נמצא רק ב-HTML של דף המוצר**:
```
API:  GET https://www.segalbaby.co.il/wp-json/wc/store/v1/products?sku=<SKU>   (או ?category=<id>)
דף:   GET <permalink>  →  לפרסר div.content-more-info (הלשוניות)
```
**חובה למשוך את שניהם** — ה-API לשדות המובנים, דף ה-HTML ל-metafields התוכן.

**היקף (עודכן 2026-07-17):** מותג **סגל בייבי (id 16) בלבד** — **11 קטגוריות**, **~150 מוצרים** לפני dedup.
קטגוריות נכללות: `beds`(37) · `dresser`(58) · `closet`(49) · `xlbabycrib`(365) · `learning-tower`(185) · `carpet`(207) · `rugs`(221) · `storage-segal-baby`(227) · `soft-close-dresser`(352) · `mattress`(65) · `beds-and-toddler`(361).
**מחוץ להיקף:** סגל קידס (214), `rooms`(47), `outlet`(174), `outlet-sale`(369), `spare-parts`(322), ו-shop-the-look — ר' **§10 פיצ'רים נדחים**.

## 1. מפתח זיהוי
**מק"ט** (`sku` מה-API ↔ `variant.sku`). אומת: 90% חפיפה (113/125), **פורמט זהה** בשני הצדדים (מספרי, למשל `3040170863`).
- מק"ט בחנות שאין באתר (12) = מוצר שסגל הוריד → לסמן לבדיקה, **לא למחוק**.
- 1 מוצר-אתר = 1 מוצר-חנות (אין איחוד וריאציות).

## 2. מיפוי השדות (שדות API)
| שדה API | → שדה Shopify | כלל |
|---|---|---|
| `name` | `product.title` | as-is (לפענח HTML entities) |
| `sku` | `variant.sku` | מפתח |
| `prices.regular_price` | **`variant.price`** (as-is) + Metafield `supplier_price` | ⭐ **לייבא כמו שהוא** ל-price; לשמור גם כ-supplier_price (reference). המרה לפי `currency_minor_unit` |
| **`stock_availability.text`** | **`variant.inventory_quantity`** | ⭐ **לסנכרן!** לפרסר "231 במלאי" → 231 · "המלאי אזל" → 0. (אומת שהיום **לא מסונכרן** — צריך לתקן) |
| `is_in_stock` | סטטוס מלאי | §5 |
| `images[].src` | `product.images[]` | למשוך ולהעלות (הורדה, לא hotlink) |
| `description` | `product.body_html` | as-is |
| `categories[]` | `product_type` + אוסף | §3 |
| `permalink` | Metafield `supplier_product_url` | עוגן |
| *(קבוע)* | `vendor` = `segal \| סגל` · `status` = **Draft** (חדש) | |
| *(לפי סוג)* | `template_suffix` | **`furniture-beds`** (מיטה/עריסה) · אחרת **`furniture-product-page`** |
| `barcode` | `variant.barcode` | סגל **לא מספק ברקוד** (ריק בכל המוצרים החיים) — אין מה למפות, לא לצפות לו |

> **ביקורת שדות מלאה (3 מוצרים חיים, כל namespace דינמית — `segal/scripts/segal_field_audit.py`):** כל שדות-הליבה + **6 metafields** (`global.title_tag`, `global.description_tag`, `custom.infoo`, `custom.view_productss`, `custom.securingg`, `custom.delivery`) מכוסים ב-100% — אף שדה בעמוד המוצר לא נשאר בלי מקור.

## 3. קטגוריות
### 3.1 קטגוריית-אתר (leaf) → `product_type`
| קטגוריה באתר | → product_type |
|---|---|
| מיטת תינוק · מזרנים | מיטות תינוק / מיטת מעבר (מעקה) |
| שידות · שידות טריקה שקטה | שידות |
| ארונות לילדים | ארונות |
| חדרי תינוקות | חדרי תינוקות |
| איחסון | אחסון |

### 3.2 `product_type` → אוסף
מיטות תינוק→מיטות תינוק · מיטת מעבר→מיטות מעבר · שידות→שידות החתלה · ארונות→ארונות לחדרי ילדים · חדרי תינוקות→חדרי תינוקות · אחסון→אחסון+אחסון ואביזרים.
> אין Smart Collections → `collects` מפורש. קטגוריה לא מזוהה → Draft + סימון.
> **rooms (חדרי תינוקות) — נדחה** (§10); השורה נשמרת במפה לכשיחזור.

## 4. Metafields — התוכן מלשוניות הדף (הליבה של סגל)
ה-metafields מגיעים מ**לשוניות הדף** (`#more-info .nav-tabs` → תוויות, `.tab-content .tab-pane` → תוכן). **מיפוי לפי שם הלשונית (label), לא לפי מיקום** (עודכן 2026-07-17 — ר' §10). התאמה עם נירמול prefix/contains. הסדר תואם את סדר הבלוקים המקורי של ה-PRD (מידע-כללי=בלוק1, פרטים-טכניים=בלוק2). אומת על **65 מוצרים** ב-11 הקטגוריות: `מידע כללי` 64/65 · `פרטים טכניים` (כולל וריאנט "פרטים טכניים, ניקוי ואזהרה") 65/65 · `Greenguard` 4/65.

> **⚠️ אי-התאמה בין תווית לשם-שדה = מכוונת.** שם הלשונית לא זהה לשם התצוגה של ה-metafield, לאורך כל המפה: `מידע כללי`→שדה "מאפיינים", `פרטים טכניים`→שדה "תיאור", `Greenguard`→שדה ההבטחה/אחריות. `custom.securingg` הוא שדה **הבטחה** (safety/assurance), ו-Greenguard היא תעודת בטיחות — לכן מיפוי `Greenguard`→`securingg` הוא תוכן-הבטחה תקין, לא באג.

| תווית לשונית (label) | → metafield |
|---|---|
| `מידע כללי` | `custom.infoo` (מאפיינים) |
| `פרטים טכניים`\* (prefix) | `custom.view_productss` (תיאור פריט) |
| `Greenguard` (case-insensitive), או תווית שמכילה `אחריות` | `custom.securingg` (הבטחה/אחריות) — אחרי הקידומת `"אחריות רחבה 5 שנים"` |
| **כל תווית אחרת (לא במפה)** | **נזרקת + נרשמת ללוג** (`tab_discarded` עם `sku`+`label`) |

> `\*` מכסה גם תוויות מורכבות ("פרטים טכניים, ניקוי ואזהרה"). לשוניות שנצפו ונזרקות כרגע: `טכנולוגיית Cloud…` · `100 לילות ניסיון` · `הוראות הרכבה`. המוצר יכול להוסיף תוויות למפה בעתיד (§10).
> **מקור `custom.securingg` בהיקף הנוכחי:** לשונית `Greenguard` (4/65) + קידומת הבוילרפלייט. לשונית `אחריות` מלאה חיה ב-rooms/kids הנדחים ותצטרף כשיתווספו.

**המרה ל-rich_text:** כל בלוק → `rich_text_field` (JSON), **כל שורה (`<br>`/`<p>`) → `paragraph` נפרד**. בולד (`<strong>`) לא נשמר — משוטח לטקסט.
**as-is:** התוכן מועתק מהבלוק כמו שהוא (בלי עריכה/קיצור). **שני נירמולים זעירים שנצפו (לא משנים את המיפוי):** (1) שורת-הכותרת בראש הבלוק עשויה להתמזג עם הפסקה הבאה (בלי מפריד פסקה); (2) בשורות מידות, `" – "` אחרי תווית ("מידות המיטה –") מופיע בחנות כ-`": "` ("מידות המיטה:"). אחרת — העתקה מדויקת.

| metafield | שם | סוג |
|---|---|---|
| `custom.infoo` | מאפיינים | rich_text ← לשונית `מידע כללי` |
| `custom.view_productss` | תיאור פריט | rich_text ← לשונית `פרטים טכניים` |
| `custom.securingg` | אחריות/הבטחה | rich_text ← "אחריות רחבה 5 שנים" + לשונית `Greenguard`/`אחריות` (Greenguard=מיפוי מכוון לשדה ההבטחה) |
| `custom.delivery` | משלוחים והחזרות | rich_text — **boilerplate קבוע** (גרסת רהיטים, verbatim) |
| `global.title_tag` | SEO כותרת | ← name |
| `global.description_tag` | SEO תיאור | ← description |

*(סנכרון: `supplier.*` — supplier_sku/price/url/stock_qty/timestamps — פנימי.)*

## 5. מלאי
| מקור | → פעולה |
|---|---|
| `stock_availability.text` = "N במלאי" | `inventory_quantity` = N |
| `stock_availability.text` = "המלאי אזל" / `is_in_stock=false` | `inventory_quantity` = 0 (Out of Stock, לא למחוק) |
> **חובה:** להגדיר `inventory_management="shopify"` ולעדכן את הכמות — הן בייבוא והן בסנכרון. (היום המלאי בחנות לא תואם לספק.)

## 6. מחיר
מייבאים את `prices.regular_price` **כמו שהוא** אל `variant.price`, ושומרים עותק כ-`supplier_price` (reference). **האוטומציה לא מחשבת נוסחה** — המשתמש משנה את מחיר-המכירה אח"כ ידנית עם הנוסחה שלו. `compare_at_price` אופציונלי (המשתמש). *(תצפית בלבד, לא בשימוש: מחיר-חנות היה ≈ ספק×0.72-0.94.)*

## 9. מה על המתכנת להשלים (Implementation TODO)
מעבר למיפוי (מכוסה למעלה) — נדרש במימוש:
1. **זיהוי מה חדש (dedup):** בכל ריצה להשוות מק"ט-ספק↔חנות → לייבא רק מק"ט שלא קיים (מק"ט חזק, 90%). מק"ט קיים → עדכון (מלאי/מחיר), לא יצירה.
2. **מקור היברידי — חובה:** למשוך גם את דף ה-HTML (בלוקי `content-more-info`), לא רק API. §0/§4.
3. **תמונות:** להוריד מהספק ולהעלות ל-Shopify CDN (לא hotlink).
4. **לאמת קבועים למוצר חדש:** אחריות "5 שנים" ו-boilerplate המשלוח — לוודא שתקפים; קטגוריית-leaf חדשה שאינה ב-§3 → Draft + סימון.
